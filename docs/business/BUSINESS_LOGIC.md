# Бизнес-логика CRM (производство, КП, склад)

**Статус:** целевая предметная модель и правила. Реализация — поэтапно: см. **[BL_PAGES_AND_DATA_MODEL.md](./BL_PAGES_AND_DATA_MODEL.md)** (страницы, входы/выходы, сущности).

**Связь с кодом:** при внедрении фичи — `libs/domain` + `docs/api/FRONTEND_CONTRACT.md`; новые API до согласования — только в `docs/api/API_FUTURE_CHECKLIST.md`.

---

## 1. Назначение

- Справочник товаров с техническими вариантами (**ProductSpecification**: категория + товар + материал + тип детали).
- **КП** с **снимком** строк (внутреннее имя: `ProposalItem` = “строка КП”), в т.ч. произвольные (custom) позиции.
- **Заказы** на производство (в т.ч. из принятого КП) с разбивкой по **видам работ** и **исполнителям**.
- **Склад / остатки**.
- **Ролевой доступ** к разделам (см. матрицу в документе по страницам).

---

## 2. Ключевые сущности (кратко)

| Сущность | Назначение |
|----------|------------|
| Organization, Client, User | Уже в домене / контракте (частично). |
| Category, Material, PartType | Справочники каталога. |
| Product | Логический товар. |
| ProductSpecification | Уникальная комбинация для спецификации и учёта. |
| Proposal, “строки КП” (`ProposalItem`) | КП и строки-снимки. |
| ObjectAddress | Адрес объекта (привязка к КП/заказу). |
| CommercialDocument | Счёт / договор (зона «Бух»). |
| WorkKind | Вид работ (лазер, сварка, …). |
| Order, OrderLine | Заказ и строки. |
| OperationAssignment | Назначение вида работ + исполнитель на строке заказа. |
| Performer | Исполнитель (часто = сотрудник / User — уточняется). |
| Warehouse, Stock | Склад и остаток. |
| StockMovement (рекомендуется) | Приход/расход/перемещение — основание для изменения Stock. |

---

## 3. Инварианты

- Отправленное / принятое КП: строки **не пересчитываются** из каталога — только **снимок** в “строке КП” (внутреннее имя: `ProposalItem`).
- Связь производства с КП: хранить **`proposalId` + `proposalItemId`** (или эквивалент «№КП–№пп»).
- История “гарантийного” восстановления: в `OrderLine` хранится snapshot **товара целиком** на момент создания заказа, чтобы можно было восстановить “из чего он состоял” и создать аналогичный товар:
  - snapshot полей товара (как минимум `name`, `sku`, и ссылки/атрибуты комплектации: `categoryId`, `partTypeId`, `materialId`, `notes`, `isActive`);
  - snapshot выбранной комплектации (mounts/functionalities): хранить `mountTypeIds`/`functionalityIds` и достаточно данных (например названия) для однозначного воссоздания состава.
- Удаление справочника, на который ссылаются КП/остатки: только **деактивация**, не hard delete (политика по согласованию).

---

## 4. Открытые решения

Перед фиксацией DTO в `FRONTEND_CONTRACT.md`: одно КП → один или несколько заказов; Performer = User или отдельный справочник; версии КП; гранулярность Stock (партия / ячейка).
