# Страницы, входные данные и модель данных (пошаговое внедрение БЛ)

**Назначение:** единая карта для фронта и бэка: какие **страницы** (фичи) вводим, какие **данные на вход** (формы, справочники, контекст), что **на выход** (экран + сохранённые сущности), какие **таблицы/сущности** нужны, если их ещё нет в контракте.

Справочник workflow-статусов и правил неизменности: `docs/business/WORKFLOW_STATUSES.md`.

**Порядок работ по архитектуре репозитория:**  
`docs/business` (требования) → при готовности к коду: `libs/domain` → `docs/api/FRONTEND_CONTRACT.md` → `core/api/*-api.service.ts` → `features/*` по **`docs/ai/FEATURE_WITH_API_PATTERN.md`**.

**Текущие маршруты приложения:** `clients`, `organizations`. Остальные пути в таблице — **предлагаемые** (`/…`).

**Краткая БЛ:** [BUSINESS_LOGIC.md](./BUSINESS_LOGIC.md).

**Добавление к фиче `products` (требование MVP):**
- у товара должен быть редактируемый набор **`mounts`** (виды монтажа), подключенный к товару;
- у товара должен быть редактируемый набор **`functionalities`** (функциональности), подключенный к товару;
- при клонировании товара сервер/БД копирует товар + его join-наборы, чтобы “новый товар” повторял комплектацию исходника (с возможностью точечных изменений).

### Фаза 3 — КП

| Маршрут (предлож.) | Фича | Вход | Выход | Основные сущности | Доп. таблицы / примечания |
|---------------------|------|------|-------|-------------------|---------------------------|
| `/proposals` | `proposals` | фильтры (клиент, статус, даты) | список КП | `Proposal (root)`, `ProposalVersion (current)` | `number`, `currentStatus`, `currentVersionNo`, `clientId`, `organizationId`, `createdByUserId`, `sentAt`, `decidedAt` |
| `/proposals/new`, `/proposals/:id` | `proposals` | клиент, организация (контекст), валюта/НДС с шапки | шапка КП + таблица позиций | `ProposalVersion`, “строки КП” (`ProposalItem` snapshot) | **Снимок в строке:** `title`, `quantity`, `unit`, `unitPrice`, `vatRate`, `lineTotal`; `productSpecificationId` **или** `isCustom` + текстовые поля + комплектация (mounts/functionalities) на момент версии |
| то же | — | выбор/создание адреса | адрес на КП | `ObjectAddress` | `proposalId` или встроенные поля в `Proposal` (`objectAddress*`) — выбрать одно; история адресов — опц. |
| `/proposals/:id` | — | действие пользователя | смена статуса | `ProposalVersion` | Журнал **`ProposalStatusHistory`** (`proposalVersionId`, `from`, `to`, `at`, `userId`) — если нужен аудит |
| `/proposals/:id` | — | шаблон | PDF | — | **`PdfGenerationJob`** или синхронный ответ URL; в backlog уже preview/pdf |

**Итог (не менять прошлое):**
Для КП “юридически/операционно фиксированным” документ считается после перевода в статус **`proposal_paid`** (name: “Оплачено”).

Тогда:
- `Proposal` и “строки КП” (`ProposalItem`) считаются **неизменяемыми** (редактирование/удаление запрещены по статусу);
- PDF генерируется **только по snapshot** строк КП;
- в “строках КП” (`ProposalItem`) хранится то, что нужно для печати “как было”, включая комплектацию товара на момент фиксации (в т.ч. выбранные `mounts`/`functionalities`, если они влияют на вид/состав печати).

**Версионирование КП (после фиксации):**
после того как КП переводится в статус `proposal_paid`, документ считается зафиксированным.

Любые исправления после `proposal_paid` делаются не “откатом статуса”, а созданием **новой версии** КП (например `01`, `02`, `03`), где:
- родительская версия (`01`) остаётся неизменной;
- новая версия создаётся как копия snapshot строк/комплектации (mounts/functionalities) и получает новый номер версии;
- в истории КП отображаются все версии, чтобы бухгалтерия/менеджер видели “что было и что стало”.

### Фаза 5 — заказы

| Маршрут (предлож.) | Фича | Вход | Выход | Основные сущности | Доп. таблицы / примечания |
|---------------------|------|------|-------|-------------------|---------------------------|
| `/production/orders` | `orders` | фильтры | список заказов | `Order` | `proposalId?`, `organizationId`, `clientId`, `status`, поля «только из КП» |
| `/production/orders/new` | — | из UI: «создать из КП» | черновик заказа | `Order`, `OrderLine` | **`sourceProposalItemLineNo`** (плюс скрытый `sourceProposalVersionId`), `productSnapshot` (денорм снимок для гарантии) |
| `/production/orders/:id` | — | позиции, количества | строки заказа | `OrderLine` | Для производства: `quantity` и `unit` (из строки КП) + гарантийный снимок товара “как было” (`productName`, `productSku`, `categoryId/categoryName`, `partTypeId/partTypeName`, `materialId/materialName`, `productNotes`, `productIsActive`) + снимок комплектации на момент заказа (`orderLineMounts`: `mountTypeId`+`mountTypeName`, `orderLineFunctionalities`: `functionalityId`+`functionalityName`) ; `lineNo`; связь с КП: `sourceProposalId` (корень) + `sourceProposalItemLineNo` (для “кода производства”); **`sourceProposalVersionId` хранится скрыто** |

---

### Идентификация товара “для производства/отгрузки”
Чтобы “код” находил концы:
- в `OrderLine` хранится `sourceProposalId` (корень КП) и `sourceProposalVersionId` (скрыто) + `sourceProposalItemLineNo` (номер позиции в КП);
- отображаемый код можно строить как `№КП` (номер root proposal, например `№0236`) + `-` + номер позиции `proposalItemLineNo` (в БД в `OrderLine.productionCode` хранится сырой вид `${proposalNumber}-${proposalItemLineNo}`).

Версионирование КП `01/02/03...` скрыто от пользователя, поэтому визуальный `№КП` остаётся `№0236` даже при нескольких версиях.

## Сводка: сущности «с нуля» для выходных данных

Не дублировать без нужды то, что уже намечено в `API_FUTURE_CHECKLIST.md` (categories, products, proposals, …). Ниже — **дополнения**, которых в старом реестре URL не было или они неявны:

| Сущность | Зачем |
|----------|--------|
| “строка КП” (`ProposalItem`) | Снимок строки КП + optional `productSpecificationId` / custom. |
| `ObjectAddress` | Адрес объекта (если не только поля в `Proposal`). |
| `ProposalStatusHistory` | Аудит статусов КП (опционально). |
| `MountType` и `ProductMount` | Виды монтажа и связь “товар ↔ виды монтажа” (для комплектации и печати). |
| `Functionality` и `ProductFunctionality` | Функциональности и связь “товар ↔ функциональности” (для комплектации и печати). |
| `CommercialDocument` + `CommercialDocumentLine` | Счета/договоры и строки (если не один PDF). |
| `Order`, `OrderLine` | Производственный заказ. |
| `sourceProposalItemLineNo` / связь №КП–№пп | Трассировка КП → заказ. |
| `WorkKind` | Виды работ. |
| `Performer` | Исполнители (если не только User). |
| `OperationAssignment` | Назначение работ на строку заказа. |
| `Warehouse`, `Stock` | Склад и количество. |
| `StockMovement` | Основание изменения остатка и отчёты. |
| `Unit` (опц.) | Единицы измерения для каталога и строк. |

| Раздел | Маршруты | Роли (из БЛ) |

## Что обновлять при внедрении фичи

1. Строка в таблице статуса в **`docs/api/API_FUTURE_CHECKLIST.md`**.
2. После согласования полей — **`FRONTEND_CONTRACT.md`** + **`libs/domain`**.
3. Чек-лист фичи в **`docs/ai/FEATURE_<NAME>_CHECKLIST.md`** (или из шаблона `FEATURE_CHECKLIST_BASE.md`).
