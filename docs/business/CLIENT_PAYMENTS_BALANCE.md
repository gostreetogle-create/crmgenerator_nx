# Заказчик: долг/оплачено и список КП/заказов (мини-финансы)

## Цель
На карточке `Client` (заказчика) и в таблицах “заказчики” показывать:
- `dolg` (долг) — сколько осталось оплатить,
- `paid` (оплачено) — сколько реально внесено,
- `advance/paid` — суммы по каждому документу в истории (КП/версии → создание заказа).

Бухгалтерия как отдельный модуль не делается: это “мини-финансы” для контроля оплат в процессе производства.

## Термины и соответствие кнопкам UI
1) Кнопка менеджера на КП: **`Оплатил`**
- Смысл: зафиксировать оплату для текущей версии КП и запустить производственный поток (после чего создаётся `Order`).
- UI-диалог: выбрать сумму для внесения.
  - ввод вручную,
  - либо кнопка “Вся сумма из КП” (предзаполнит сумму = обязательство документа).
- После сохранения:
  - `paidSum` увеличивается на введённую сумму,
  - `dolg` становится `totalCommitted - paidSum` (если ввели всю сумму — `dolg = 0`).

2) Кнопка на КП или в списке оплат: **`Аванс`**
- Смысл: зафиксировать частичное поступление денег (т.е. `amount < totalCommitted`).
- Технически в MVP может быть тем же действием, что и `Оплатил`, но тип операции помечается как `advance`.

3) Последующие платежи
- Если нужно: “доплата/остаток” реализуется как ещё одна запись факта оплаты.
- Когда `paid >= totalCommitted`, `dolg` становится 0.

## Как вычислять суммы
Для **каждой версии КП, по которой зафиксированы платежи** (обычно после первого `proposal_paid`):
- `totalCommitted` — сумма обязательства (заказа), зафиксированная на момент события “Оплатил”.
- `paidSum` — сумма всех фактов поступления денег по этому документу.
- `debt = totalCommitted - paidSum` (не ниже 0, при перерасчетах — min/max в бэке).

Для **заказчика (Client)**:
- `paid = sum(paidSum по всем документам заказчика)`;
- `dolg = sum(totalCommitted по документам заказчика) - paid`.

## Версионирование КП и влияние на деньги
- Поскольку после `proposal_paid` прошлое неизменяемо, фиксирование денег (создание обязательства/счёта) привязывается к **конкретной версии КП**.
- Если после `proposal_paid` делается новая версия КП (02/03/...), то обязательство и платежи относятся к соответствующей версии.

## Данные, которые нужно хранить в БД
### 1) “Фиксация обязательства” (якорь)
На уровне `ProposalVersion` хранить:
- `proposalVersionId` (ссылка),
- `totalCommitted` (сумма обязательства на момент фиксации),
- `committedAt` (дата фиксации).

В MVP `totalCommitted` вычисляется как сумма “заказа” (итог КП на момент создания заказа).

### 2) Факты оплаты (транзакции)
Таблица `ProposalPayment` (или `PaymentTransaction`) хранит:
- `id`
- `proposalVersionId`
- `amount` (сколько поступило)
- `paymentType`: `advance` | `rest` | `full` (обычно определяется по введённой сумме относительно `totalCommitted`)
- `paidAt` (когда поступило)
- опционально: `comment`, `paymentDocRef` (реквизит/номер платежки) — если появится позже.

## Какие эндпоинты нужны для реализации
1) На уровне КП:
- `POST /api/proposals/:id/payments`
  - принимает факт оплаты для **текущей версии** или конкретной версии (в MVP можно только текущей)
  - возвращает updated `paid`/`debt` по документу
  - вход: `amount` и (опционально) `paymentType` или признак “вся сумма”
  - поведение:
    - если платеж создаётся впервые для версии — одновременно фиксируется `totalCommitted` и (по вашему workflow) переводит версию в `proposal_paid` и запускает создание `Order`.

2) На уровне заказчика:
- `GET /api/clients/:id/balance`
  - возвращает агрегаты: `totalCommitted`, `paid`, `debt`

- `GET /api/clients/:id/proposals`
  - возвращает список документов (КП/версии) с полями:
    - `proposalId` / `proposalVersionId`,
    - `versionNo`,
    - `totalCommitted`,
    - `paidSum`,
    - `debt`,
    - `status` (key/name текущей/зафиксированной версии).

## Правила неизменности
- Факт оплаты (транзакция) после записи считается неизменяемым.
- Коррекция делается новой транзакцией (например `amount` со знаком “-” для возврата) только для роли админ/бух (если понадобится позже).
