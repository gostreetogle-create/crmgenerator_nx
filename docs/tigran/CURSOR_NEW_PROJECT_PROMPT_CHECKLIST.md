# Чеклист Тиграна: новый проект в Cursor

Цель: быстро развернуть одинаковый стандарт документации и правил ИИ для нового приложения.

## 1) Минимальная структура

- [ ] Создать `docs/README.md` и `docs/START_HERE.md` (единая точка входа).
- [ ] Создать `docs/ai`, `docs/api`, `docs/business`, `docs/tigran`.
- [ ] Перенести базовые шаблоны: `FEATURE_CHECKLIST_BASE.md`, `DOCS_SYNC_RULES.md`, `PROMPTS.md`.

## 2) Cursor rules (.mdc)

- [ ] Создать `.cursor/rules/<project>-global-prompts.mdc` с `alwaysApply: true`.
- [ ] Создать `.cursor/rules/<project>-ai.mdc` с `globs` под код и docs проекта.
- [ ] В обоих файлах указать единый старт чтения: `docs/START_HERE.md`.

## 3) Что писать в глобальный prompt

- [ ] Порядок чтения документов (архитектура -> playbook -> контракт -> checklist).
- [ ] Нельзя нарушать слои и импортные ограничения.
- [ ] Любая правка кода = синхронизация документации.
- [ ] Для каждой фичи обязателен checklist и факт выполнения проверок.

## 4) Что писать в project prompt

- [ ] Технологический стек проекта.
- [ ] Путь к командам проверок (`lint`, `build`, `test`, `smoke`).
- [ ] Правила по UI-kit, domain, API.
- [ ] Правила коммитов и PR (если уже формализованы).

## 5) Проверка зрелости документации (DoD)

- [ ] Есть один стартовый файл и он реально покрывает маршрутизацию по задачам.
- [ ] API “истина” отделена от backlog (`CONTRACT` vs `FUTURE`).
- [ ] Есть feature-checklist и архив завершенных итераций.
- [ ] Есть индекс архитектурных артикулов / стандартов.
- [ ] Новому участнику понятно, что читать первым, вторым, третьим.

## 6) Шаблон внедрения в первый день

1. Создать каркас docs.
2. Создать 2 `.mdc` правила (global + project).
3. Описать MVP фичу в checklist.
4. Синхронизировать API контракт.
5. Прогнать `lint/build`.

---

**Результат:** новый проект запускается по повторяемому шаблону, без потери контекста и без “ручного хаоса”.
