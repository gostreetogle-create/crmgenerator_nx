# Архитектура фронтенда (NX)

**Важно для ИИ и разработчика:**  
Любое изменение структуры, слоёв, импортов, фич и публичных API должно отражаться в этом файле.

## Цель
Поддерживать предсказуемый и масштабируемый Angular/Nx фронтенд:  
- переиспользуемый UI в `ui-kit`  
- бизнес-логика в `features`  
- чёткие границы слоёв  
- единый дизайн через `design-tokens`  
- минимальный риск "архитектурного мусора"

## Слои и ответственность

### 1) `apps/web` (composition layer)
- Точки входа приложения (`main.ts`, `app.config.ts`, роутинг, layout).
- Связывает фичи между собой, но не хранит доменную бизнес-логику.
- Не содержит дублирующие UI-компоненты, если они уже есть в `ui-kit`.

#### `apps/web/src/app/core/api` (HTTP к бэкенду)
- **`environments/`** — `environment.ts` / `environment.development.ts`: `production`, **`apiBaseUrl`** (пусто = без удалённого API, только текущие feature-сервисы).
- Подключение в **`app.config.ts`**: `APP_ENVIRONMENT`, `provideHttpClient(withFetch(), withInterceptors([apiBaseUrlInterceptor]))`.
- **`OrganizationsApiService`**, **`ClientsApiService`** — тонкие обёртки над `HttpClient` по путям из **`docs/api/FRONTEND_CONTRACT.md`** (только то, что уже внедрено); идеи на потом — **`docs/api/API_FUTURE_CHECKLIST.md`** (перед кодом — согласовать и перенести в контракт).
- **`OrganizationsService` / `ClientsService`** при непустом **`apiBaseUrl`** используют эти API-сервисы (загрузка списка + CRUD); иначе — сид и **`localStorage`**.
- **Норматив для любой новой такой фичи:** **`docs/ai/FEATURE_WITH_API_PATTERN.md`** — пошагово: `domain` → `XxxApiService` → `XxxService` (сигналы, оптимизм, кэш) → страница с баннерами и тесты.
- Фичи **не** дублируют сырые URL: относительные строки для `HttpClient`, база задаётся один раз в `environment`.

### 2) `apps/web/src/app/features/*` (domain feature layer)
- Каждая папка — отдельная предметная фича (`clients`, `organizations`, `products`...).
- Внутри фичи: page/container, формы, локальные модели/мапперы, feature-сервис.
- Feature не импортирует другую feature напрямую.

### 3) `libs/ui-kit` (presentational UI layer)
- Только "глупые" переиспользуемые компоненты (Button, Input, Card, ConfirmDialog...).
- Никакой доменной логики, HTTP, роутинга, knowledge о фичах.
- API через `input/output`; состояние только UI-уровня (open/loading/disabled).
- **Живой каталог и правила единого стиля:** `docs/ai/UI_CATALOG_AND_DESIGN_SYSTEM.md` (кнопка **UI** в шапке приложения). Новый компонент в `ui-kit` или новый общий CRUD-паттерн → обновить каталог в `features/ui-catalog/`.

### 4) `libs/design-tokens` (design system layer)
- Единственный источник токенов: цвета, spacing, тени, типографика, radius.
- Любая визуальная настройка через `var(--...)`, без хардкода.

### 5) `libs/shared` (cross-feature logic layer)
- Общие типы, утилиты, guards, pure функции.
- Без Angular UI-компонентов.

## Smart и Dumb компоненты (обязательное правило)

### Smart (container) компоненты
- Расположение: только `features/*`.
- Отвечают за: получение данных, orchestration, сигналы состояния, вызовы сервисов, роутинг.
- Могут знать DTO/модели конкретной фичи.

### Dumb (presentational) компоненты
- Расположение: преимущественно `libs/ui-kit` (или локально в фиче, если компонент 100% специфичен и не переиспользуется).
- Отвечают только за отображение и события `input/output`.
- Не делают HTTP, не ходят в `Router`, не хранят бизнес-правила.

Правило потока: **Smart -> Dumb через `input`**, **Dumb -> Smart через `output`**.

## Контракт API для фронта (ведение вручную)

- **Истина по API, уже внедрённому во фронте:** `docs/api/FRONTEND_CONTRACT.md` — эндпоинты и поля под текущий код (`domain`, `core/api`).
- **Черновик / backlog API:** `docs/api/API_FUTURE_CHECKLIST.md` — не считать контрактом до переноса в `FRONTEND_CONTRACT.md`; **перед внедрением** уточнять поля.
- **Когда обновлять:** при добавлении фичи с данными с сервера, при изменении полей форм/моделей, при появлении новых требований от бэка (загрузки файлов, пагинация, фильтры, ошибки).
- **Цель:** договорённости по **текущему** коду — **`FRONTEND_CONTRACT.md`** + **`libs/domain`** + **`core/api`**. План на будущее ведётся в **`API_FUTURE_CHECKLIST.md`** до промоции в контракт. Старый `docs/Пример полей схем бд/api.ts` не обязателен — сверка с чеклистом, затем можно удалить папку.
- **Бизнес-требования и поэтапный план страниц / данных:** **`docs/business/BL_PAGES_AND_DATA_MODEL.md`** (+ кратко **`docs/business/BUSINESS_LOGIC.md`**). При новой фиче сверяться с матрицей фаз и доп. сущностями до проектирования DTO.

### Производство / Заказы (`Order`)

При внедрении страницы производства (`/production/orders*`) считать “истиной” **табличные snapshot’ы**, а не PDF:
- фронт получает данные через `GET /orders` и `GET /orders/:id` (см. `docs/api/FRONTEND_CONTRACT.md`);
- восстановление “аналогичного товара для гарантии” делается по `OrderLine.productSnapshot` + выбранной комплектации `mounts/functionalities`;
- для производства по строке обязательны `OrderLine.quantity` и `OrderLine.unit` (цена не хранится/не нужна);
- PDF формируется по snapshot, но данные состояния/истории не зависят от того, хранится ли PDF-файл на диске.

## Эталон: фича с CRUD и опциональным HTTP

Все новые data-фичи (список + формы + опциональный бэк) делай **в одном стиле** с `organizations` и `clients`:

1. Читай **`docs/ai/FEATURE_WITH_API_PATTERN.md`** — там таблица эталонных файлов, чек-лист по слоям, запреты и требования к тестам.
2. Не отклоняйся без причины: иначе усложняется сопровождение и онбординг ИИ/людей.

Кратко: **`libs/domain`** → **`docs/api/FRONTEND_CONTRACT.md`** → **`core/api/*-api.service.ts`** → **`features/<name>/*.service.ts`** (сигналы `listLoading` / `listLoadError` / `mutationError`, LS-кэш при remote, оптимистичный CRUD) → **страница** с баннерами (`protected` inject сервиса).

## Точки входа и публичный API

- Каждый reusable модуль имеет `index.ts` как публичный entry point.
- Импортировать только через публичный API, не через глубокие пути файлов.
- Для `ui-kit`:
  - внешний импорт из app: `@ui-kit/button`, `@ui-kit/input`, `@ui-kit/card`, `@ui-kit/confirm-dialog`
  - внутренние импорты внутри `ui-kit`: относительные (`../button`, `../card`)
- Запрещены deep-importы вида `libs/.../src/lib/.../file.ts`.

## Правила переиспользуемости

- Если компонент нужен в 2+ фичах -> переносим в `libs/ui-kit` (UI) или `libs/shared` (logic/types).
- Если компонент фиче-специфичен и не ожидается reuse -> оставляем в feature.
- Не дублировать одинаковые кнопки/инпуты/карточки в `features`.
- Любые изменения базового UI сначала отражаются в Component Catalog.

## Дизайн и стилизация

- Для ui-компонентов `ViewEncapsulation.None`.
- Классы применяются к реальным DOM-элементам, не только host.
- В `.scss` компонентов обязательный импорт токенов: `@use "tokens";`
- Цвета/тени/шрифты/spacing/radius: только `var(--...)` из `design-tokens`.
- Запрещён визуальный хардкод (`#...`, `rgba(...)`, fixed shadow values), кроме обоснованных случаев в токенах.

### Стили форм в `features/*` при `ViewEncapsulation.None`

Формы в фичах (например `*-form.component`) часто используют `ViewEncapsulation.None` по тому же принципу, что и `ui-kit`. Тогда стили из `.scss` попадают в **глобальную** таблицу стилей страницы.

**Обязательно:**

- Корневой уникальный класс на корне шаблона формы (например `.client-form`, `.org-form`, `.<feature>-form`).
- Все селекторы в SCSS этой формы вложены под этот корень: `.org-form .field`, `.org-form .notes-area`, а не «голые» `.field` / `.notes-area` / `.form-actions` без родителя.

**Запрещено:**

- Повторять в разных фичах одинаковые **глобальные** имена классов без уникального родителя (иначе последняя загруженная фича перетирает визуал другой — «цвета не соответствуют», поломанные отступы и т.д.).

**Альтернатива (если не хотите помнить про вложенность):** оставить для формы фичи дефолтную инкапсуляцию (`Emulated`), тогда конфликтов между фичами по именам классов не будет; согласуйте с командой, если отходите от паттерна существующих форм.

**Проверка при вводе новой фичи с формой:** после `nx serve web` открыть **и** старую фичу (например `/clients`), **и** новую — убедиться, что модалки/поля выглядят как до изменений.

### Палитры тем (единый источник)
- Светлая тема (`:root`) — образ **«лев»**: тёплые песочные нейтрали, золото/грива как `primary`, рыжевато-коричневый как `secondary`.
- Тёмная тема (`.dark`) — образ **«пантера»**: уголь/ночь, сталь как `secondary`, янтарный блик как `primary`/`outline`.
- Все правки сочетаний цветов — **только** в `libs/design-tokens/src/styles/tokens.scss`; артикулы `col-*` в Component Catalog отражают эти же CSS-переменные.

### Переключение темы (light/dark)
- Тема управляется классом `dark` на `document.documentElement`.
- Значение темы сохраняется в `localStorage` (ключ `theme`), чтобы выбор восстанавливался после перезагрузки.
- Любые UI-изменения должны опираться на `design-tokens` (разные значения в `:root` и `.dark`), а не на прямую подстановку цветов в фичах.

### Обязательные поля в ui-kit формах
- Для обязательных полей в `ui-kit` используйте `app-input`:
  - управляйте ошибкой через `error`
  - **скрывайте текст ошибки** через `[showErrorText]="false"` (оставляем только красную рамку и красную надпись `label`)
- Для не обязательных полей (например, `email`/`phone` при ошибке формата) оставляйте `showErrorText` по умолчанию (`true`), чтобы пользователь видел текст причины.

## Модальные окна и формы с многими разделами (1C / enterprise-стилистика)

**Назначение:** зафиксировать подход, которым доведена форма **«Новая организация»** (`organizations`), чтобы следующие диалоги с большим числом полей **не превращались в бесконечный скролл**, **не прыгали по размеру** и оставались **компактными и структурированными** — в духе плотных учётных интерфейсов (ориентир: 1С: много групп полей, мало «воздуха», ясная иерархия).

### Когда применять
- В модалке **больше ~2–3 логических блоков** или форма не помещается на экран без ощутимого скролла **всего** диалога.
- Пользователь переключает контекст по смыслу (реквизиты / адреса / контакты / файлы и т.д.) — как отдельные «страницы» внутри одного окна.
- Для таких форм **обязателен стиль "как в организациях"**: вкладки, фиксированная геометрия модалки, скролл только внутри активной вкладки.

### Обязательные паттерны

1. **Вкладки вместо одной колонны**  
   - Одна `<form>`, одно состояние; видимость разделов переключается вкладками (`signal` + `@if` / `@switch`).  
   - Группировать смыслы: например *Реквизиты*, *Адреса и банк*, *Контакты и руководитель*, *Файлы и настройки* (3–6 вкладок — норма).  
   - Не плодить отдельные модалки на каждый блок без необходимости.

2. **Фиксированная геометрия модалки**  
   - Задать у контейнера диалога (например `.modal-panel` фичи) **`height` и `min-height`** в виде `min(<целевая высота в px>, <доля vh>)`, плюс **`max-height`** по `vh`, чтобы окно **не меняло высоту при смене вкладки**.  
   - Ширина — `min(px, vw)` по необходимости; **`overflow: hidden`** у панели, чтобы скроллился не весь диалог.

3. **Где скроллить**  
   - Прокрутка только у **области контента активной вкладки** (`overflow-y: auto`, `min-height: 0` в flex-цепочке).  
   - **Кнопки** («Отмена», «Сохранить») — **внизу формы**, `flex-shrink: 0`, визуально отделены (например `border-top`).

4. **Flex-цепочка (кратко)**  
   `modal-panel` (column, заданная высота) → хост формы / карточка → `card-body` → `form` (`flex: 1`, `min-height: 0`) → ряд вкладок (`flex-shrink: 0`) → **панель вкладки** (`flex: 1`, `min-height: 0`, скролл) → **действия** (`flex-shrink: 0`).

5. **Компоновка внутри вкладки**  
   - Сетка **`grid`** с **`column-gap` / `row-gap`** из токенов; между двумя соседними колонками (особенно «файлы | настройки») при необходимости **`column-gap: clamp(...)`** и **доп. padding к шву** у соседних `fieldset`, чтобы блоки **не визуально слипались**.

6. **Плотность и поля**  
   - Высота и ритм полей — через **`ui-kit`** (`app-input` и т.д.), отступы секций — **`var(--spacing-*)`**.  
   - Секции оформлять как **группы с рамкой** (`fieldset` + `legend` или эквивалент), без лишней декоративности — ближе к **компактной учётной форме**, не к маркетинговому лендингу.

### Чек-лист для новой модалки с многими разделами
- [ ] Вкладки или иной явный разбиение контента без одной бесконечной колонны  
- [ ] У панели диалога зафиксированы высота (и при необходимости ширина) — нет «прыжка» при смене вкладки  
- [ ] Скролл только внутри области вкладки; действия всегда видны внизу  
- [ ] Явные горизонтальные зазоры между соседними колонками блоков  
- [ ] Стили формы под уникальным корнем и вложенные селекторы при `ViewEncapsulation.None` (см. выше)

### Каскад модалок (родитель + quick-add) — обязательный паттерн

Назначение: сценарий вида `Редактировать товар` -> `+` (`Новая категория/тип/материал`) должен закрывать только верхний слой.

Обязательные правила:

1. Верхний `+`-диалог рендерится отдельным оверлеем через `@ui-kit/quick-add-dialog` и находится выше родительской формы по слою.
2. Вложенная форма quick-add изолирует submit на нативном событии:
   - `<form (submit)="onNgSubmit($event)">`
   - внутри обработчика обязательно `event.preventDefault()` и `event.stopPropagation()`.
3. Для вложенных quick-add форм не использовать `(ngSubmit)` без явного импорта и проверки `FormsModule`; базовый стандарт проекта для этого кейса — обычный `(submit)`.
4. Родительская форма в период quick-add взаимодействий должна быть заблокирована от кликов/submit (через `interactionLocked` и/или `pointer-events: none` на панели родителя).
5. Кнопка `Сохранить` в родительской форме отключается при невалидном состоянии (`hasErrors`) и не может быть «активной, но бездействующей».

Минимальная ручная проверка:
- открыть родительскую модалку -> открыть `+`-диалог -> `Сохранить` во вложенной форме;
- ожидаемо: закрывается только верхняя модалка, родитель остаётся открыт;
- после заполнения обязательных полей в родителе: `Сохранить` закрывает родителя.

### Стандарт размеров модалок (обязательный)

Единая шкала размеров задаётся в `catalog-crud-patterns.scss` и используется во всех CRUD-фичах:

- `modal-panel--sm` — короткие быстрые формы (quick-add, 1-3 поля)
- `modal-panel--md` — обычные CRUD-формы справочников
- `modal-panel--lg` — расширенные формы (несколько секций, но без перегруза)
- `modal-panel--xl` — большие формы с вкладками/многими разделами

Назначение по текущему проекту:
- `categories`, `materials`, `part-types` -> `modal-panel--md`
- `clients` -> `modal-panel--lg`
- `organizations`, `products` -> `modal-panel--xl`
- `@ui-kit/quick-add-dialog` по умолчанию `size="sm"` (повышать до `md/lg` только при реальной необходимости)

Запрещено:
- задавать произвольные локальные размеры модалок в каждой фиче без согласования со шкалой `sm/md/lg/xl`;

### Layout baseline страниц (чтобы не было «пустого полотна»)

Для страниц CRUD обязателен единый базовый layout:

1. Хедер приложения — стабильный (`sticky`) с фиксированной визуальной высотой.
2. Главная рабочая область (`app-main`) имеет `min-height` на viewport и ограничение по максимальной ширине контейнера.
3. Контент CRUD-страницы (`catalog-crud-page`) имеет минимальную рабочую высоту; карточка списка заполняет доступную область по высоте.
4. Нельзя оставлять страницы в состоянии, где контент занимает только верхнюю часть экрана, а основной viewport визуально пуст.

### UX-стандарты для CRUD-таблиц (обязательно)

1. В строке действий таблицы использовать `app-button`, а не локальные `link-btn`:
   - просмотр: `variant="ghost"`
   - редактирование: `variant="secondary"`
   - удаление: `variant="danger"`
2. Для состояния пустого списка/поиска использовать явную строку `@empty` в `<tbody>` с понятным текстом следующего действия.
3. Действия баннеров ошибок (`Скрыть`, `Закрыть`) делать через `app-button` (`ghost`/`secondary`), чтобы поведение было единообразным во всех фичах.
4. Для составных экранов (пример: `products` с боковыми колонками) состояния `loading/error/retry` обязательны **не только** для центральной таблицы, но и для side-panels.

### Навигация app-shell (обязательно)

- Основное меню в шапке строится на `routerLink` + `routerLinkActive`, а не на программной навигации по клику.
- Пункты меню должны быть семантическими ссылками (`<a>`), чтобы корректно работали клавиатура, deep-link и state активного маршрута.

## Guardrails по импортам

- `features/*` -> можно: `ui-kit`, `shared`, `design-tokens`; нельзя: другие `features/*`.
- `ui-kit` -> можно: `design-tokens`; нельзя: `features/*`.
- `shared` -> нельзя зависеть от `features/*` и `ui-kit`.
- `apps/web/core` -> может оркестрировать feature-маршруты и app-level провайдеры, но не дублирует feature-сервисы.

## Component Catalog

- Кнопка `UI` в хедере открывает каталог компонентов.
- Каталог — источник правды по артикулам и визуалу (`b-01`, `b-02`, `i-01`, `c-01`...).
- Любая правка ui-kit проверяется в каталоге и в реальных экранах.

## Обязательная проверка перед коммитом

1. `npm run ui:lint`
2. `nx serve web` + hard refresh (`Ctrl+F5`) для визуальной проверки
3. Проверка, что нет deep-import и нет новых дублирующих компонентов
4. Обновление `docs/ai/ARCHITECTURE.md`, если менялась структура или правила  
5. При изменениях API/полей сущностей — актуализировать **`docs/api/FRONTEND_CONTRACT.md`**; новые идеи до согласования — в **`docs/api/API_FUTURE_CHECKLIST.md`**

## Жёсткие запреты

- Нет локальных дублей `button/input/card/confirm-dialog` внутри `features`.
- Нет импортов из feature в feature.
- Нет deep-import в обход `index.ts`.
- Нет хардкода дизайн-значений вне `design-tokens`.
- Нет создания архитектурных сущностей без обновления этого документа.
- Нет «голых» глобальных селекторов в SCSS форм фич при `ViewEncapsulation.None` без уникального корневого класса (см. раздел «Стили форм в features» выше).
