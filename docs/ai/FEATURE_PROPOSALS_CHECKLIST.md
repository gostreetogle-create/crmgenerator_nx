# Фича `proposals` — КП (чеклист состояния)

## Summary
Фича управления КП (proposal): список, просмотр, изменение статуса, блокировка редактирования после перехода в статус `proposal_paid`, функция версионирования КП и печать PDF (preview/pdf) по snapshot.

## Definition of MVP (первая итерация)
Пользователь может:
- открыть список КП и карточку/детали КП;
- на стороне менеджера менять статус КП до перехода в `proposal_paid`;
- после того как **текущая версия** КП перешла в `proposal_paid`, кнопки `редактировать` и `удалить` скрываются/становятся недоступны по статусу (даже если пользователь открыл страницу “бухгалтера” с соответствующим маршрутом);
- сгенерировать preview/pdf для КП, которое печатается по snapshot строк КП (комплектация товара, mounts/functionalities, не “плывёт” при изменениях справочников);
- создать новую **версию** КП (02/03/...) после `proposal_paid`, копируя позиции как snapshot, чтобы старое “прошлое” осталось неизменным, а менеджер мог править “новое будущее”.

## Non-goals
- бухгалтерия (счета, проводки) не делаем в этой итерации;
- импорт из Excel и загрузка больших файлов в MVP не делаем;
- сложные фильтры/пагинация “как в enterprise” не делаем (достаточно MVP-выборок по параметрам).

## Implementation checklist

### Архитектура и слои
- [ ] Структура фичи в `apps/web/src/app/features/proposals` соответствует паттерну `FEATURE_WITH_API_PATTERN.md`.
- [ ] В UI не делаются HTTP-вызовы напрямую, используется `*ApiService` в `apps/web/src/app/core/api`.

### Статусы, блокировка редактирования, RBAC
- [ ] В БД статусы создаются заранее и завязаны на стабильный `statusId`/`key`; редактируется только `name`.
- [ ] Для КП используются статусы `proposal_waiting`, `proposal_approved`, `proposal_paid`.
- [ ] Правило блокировки: когда **текущая версия** КП в `proposal_paid`, операции `edit` и `delete` запрещены на уровне UI и на уровне API.
- [ ] Бизнес-правило “не менять прошлое”: удаление корня КП запрещено, если среди версий есть хотя бы одна `proposal_paid`.
- [ ] Правило переходов: бэк валидирует допустимость перехода статуса (например, нельзя “назад” из `proposal_paid` без отдельного админ-пути).

### Snapshot и неизменность “прошлого”
- [ ] При переходе в `proposal_paid` создаётся snapshot данных КП для печати (в `ProposalItem` сохраняются поля/комплектация, влияющие на вид PDF; включая выбранные mounts/functionalities либо уже разложенные печатные поля).
- [ ] PDF генерируется строго по snapshot, а не подтягивает “живые” справочники/комплектации.

### Версионирование КП
- [ ] Реализовать UI действие “Создать версию” (например 02/03/...) для КП после `proposal_paid`.
- [ ] Реализовать backend endpoint `POST /proposals/:id/versions` (или аналог), который создаёт новую версию КП и копирует позиции как snapshot.
- [ ] Новая версия получает редактируемый статус (предложение: `proposal_waiting` или аналог для менеджера).
- [ ] UI и API работают с **текущей версией** КП (current version): редактирование/изменение статуса применяются к текущей версии.
- [ ] Отображается история версий: список версий `01/02/03/...` доступен для просмотра (печать/preview по конкретной версии).

### PDF генерация (не грузим сайт)
- [ ] Реализовать асинхронную генерацию PDF: `POST` создаёт job, `GET` отдаёт ссылку/статус или итоговую ссылку.
- [ ] Кэшировать результат по snapshot (не генерировать повторно, если snapshot не менялся).
- [ ] Ограничить параллельность воркеров (1–2) для защиты CPU/RAM.

### UI / UX
- [ ] В списке КП отображаются минимум `number`, `status`, `clientId` (или имя клиента).
- [ ] На странице КП кнопки `редактировать/удалить` скрываются/деактивируются при статусе `proposal_paid` **текущей версии**.
- [ ] На странице КП пользователь видит “какая версия сейчас актуальна” (редактируется) и может перейти в просмотр предыдущих версий.

## Data/validation notes
- [ ] Входные данные в позициях КП валидируются (кол-во, цены, обязательные поля строки).
- [ ] Поля `mounts` и `functionalities` на уровне snapshot сохраняются так, чтобы PDF оставался неизменным после правок товара/справочников.

## Integration points
- [ ] Подключение real API в `*ApiService` и `FRONTEND_CONTRACT.md` только после согласования полей.
