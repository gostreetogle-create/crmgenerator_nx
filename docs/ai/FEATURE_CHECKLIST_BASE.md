# Чек-лист состояния фич (Feature State)

См. также: `docs/ai/AI_PROJECT_PLAYBOOK.md`, **`docs/PROMPTS.md`** (готовые промпты P1–P8).

## Зачем это нужно
Этот документ — “единый источник правды” по каждой фиче: что уже сделано, что в процессе, что заблокировано, и какие решения/ограничения сейчас действуют.  
Цель: ускорить разработку и снизить риск “переизобретений” и рассинхронизации между людьми.

## Где хранить
- База шаблона: `docs/ai/FEATURE_CHECKLIST_BASE.md`
- Чек-листы конкретных фич: `docs/ai/FEATURE_<FEATURE_NAME>_CHECKLIST.md`
  - пример: `docs/ai/FEATURE_CLIENTS_CHECKLIST.md`
- Заархивированные чек-листы (завершено и не в работе): `docs/ai/archive/FEATURE_<FEATURE_NAME>_CHECKLIST.md`

## Когда и как редактировать
Чек-лист должен обновляться:
- перед началом работы (опиши MVP/границы/ожидаемый результат)
- после каждого значимого изменения в фиче (функция добавлена/улучшена/перенесена)
- когда что-то не доделали (оставь статус `blocked/partial` и причину)
- когда откатили изменения (добавь запись в `History` и верни чекбокс в актуальное состояние)

Практическое правило: если в коде фичи что-то поменялось, документ в той же сессии тоже должен быть обновлён.

Если фича затрагивает **данные с бэка**: после согласования полей — **`docs/api/FRONTEND_CONTRACT.md`** + `libs/domain`; до этого черновик — **`docs/api/API_FUTURE_CHECKLIST.md`** (см. `ARCHITECTURE.md`).

**Если фича = список + CRUD + опциональный HTTP (как организации/клиенты):** обязательно следуй **`docs/ai/FEATURE_WITH_API_PATTERN.md`** и в `Links` чек-листа укажи этот файл как норматив. Код пиши **однотипно** с эталонами из таблицы в том документе.

## Процесс завершения (архивация) — “всё готово”
Фича считается завершённой и готовой к архивации, когда одновременно выполнены условия:
- В `Implementation checklist` нет пунктов `- [!]` (blocked) и нет `- [-]` (частично) по критериям MVP.
- В `Risks / open questions` нет критичных открытых вопросов (или они явно перенесены в backlog следующей итерации).
- В `Tests & QA` отмечены хотя бы “минимальные проверки” (ручные или автоматизированные) для сценариев MVP.

После этого файл чек-листа:
1. Помечается в конце документа как “Готово / в архиве”.
2. Перемещается в `docs/ai/archive/` (и при необходимости переименовывается по дате/версии).

Важно: чтобы избежать повторного создания, в заархивированной версии чек-листа оставляй финальную строку-маркер “Готово / в архиве” (см. шаблон внизу).

## Процесс при создании новой фичи (для человека и ИИ)
Для каждой новой фичи нужно пройти один и тот же путь:
1. Создать файл `docs/ai/FEATURE_<FEATURE_NAME>_CHECKLIST.md` из этого шаблона.
2. Заполнить `Summary`, `Links`, `Definition of MVP`, `Non-goals` и `Implementation checklist`.
3. По мере работ обновлять чекбоксы/статусы и добавлять записи в `History`.
4. Когда всё критерии MVP закрыты — выполнить “Процесс завершения (архивация)”.

Автоматизация: этот процесс “агент-совместимый”.
Если ИИ/агент имеет доступ к файловой системе, он должен:
- перед стартом проверить, нет ли уже `FEATURE_<FEATURE_NAME>_CHECKLIST.md` в архиве
- если есть заархивированная версия — не дублировать, а создать новый чек-лист только если это *новая итерация/версия* (и тогда указать связь в `History` и `Risks / open questions`)

Сценарии, которые должен уметь “автопроцесс”:
1. Старт новой фичи
   - ИИ определяет `FEATURE_NAME`
   - Создаёт `docs/ai/FEATURE_<FEATURE_NAME>_CHECKLIST.md` из шаблона
   - Вносит первый блок `History` (“инициализация чек-листа”)
2. Разработка итерациями
   - После каждого изменения состояния MVP ИИ обновляет `Implementation checklist`
   - В конце итерации добавляет 1-3 строки в `History`
3. Частичный прогресс / блокер
   - Если критерии MVP не закрыты: помечается `- [!]` или `- [-]` + причина в `Risks / open questions`
4. Откат
   - Если код/функциональность откатили: ИИ возвращает чекбоксы в актуальное состояние
   - Добавляет запись в `History` (“откатлено: причина/что вернули”)
5. Завершение
   - ИИ подтверждает выполнение критериев (нет `- [!]` и нет `- [-]` в MVP)
   - Заполняет `Completion & Archiving` и выставляет финальную строку-маркер
   - Перемещает файл в `docs/ai/archive/`
6. Повторное возвращение фичи после архива
   - ИИ видит архивную версию и создаёт *новый* файл для новой итерации
   - В `History` указывает ссылку на архивную версию

## Обязательный формат файла фичи
В каждом `FEATURE_<FEATURE_NAME>_CHECKLIST.md` должны быть ключевые секции:

1. `Summary` — 3-6 строк: что за фича и где находится.
2. `Links` — ссылки на `ARCHITECTURE.md` и на конкретные файлы/роуты (минимум: page, service, модели).
3. `Definition of MVP` — что считается “готово” для первой итерации.
4. `Non-goals` — что сознательно НЕ делаем в текущей фазе.
5. `Implementation checklist` — чекбоксы с критериями готовности.
   - Используй статусы:
     - `- [x]` выполнено
     - `- [!]` заблокировано (укажи причину)
     - `- [-]` частично (укажи, что именно отсутствует)
     - `- [ ]` не начато
6. `Data/validation notes` — правила валидации/нормализации (если есть).
7. `Integration points` — какие места будут меняться при подключении реального бэка/API (repository, DTO, persistence). Для стандартной data-фичи укажи соответствие **`FEATURE_WITH_API_PATTERN.md`** (уже реализовано / отличия).
8. `Tests & QA` — какие тесты/проверки планируются и что уже есть.
9. `Risks / open questions` — риски и вопросы.
10. `History` — краткие записи “когда/что решили/что изменили”.
11. `Completion & Archiving` — финальный маркер “готово/в архиве” (см. ниже).

## Как отмечать “критерий готовности”
Каждый пункт чек-листа должен быть сформулирован так, чтобы можно было проверить глазами или короткими действиями:
- “можно добавить/редактировать/удалить…”
- “пользователь видит ошибки в форме…”
- “фильтры сохраняются в query params и доступны по прямой ссылке…”
- “данные переживают перезагрузку страницы…”

## Минимизируем бюрократию
Чек-лист не должен разрастаться:  
если пункт стал неактуален — пометь его как `- [x]`/`- [-]`, или удаляй только когда уверенно понимаешь, что он больше не понадобится.

## Completion & Archiving
Финальный блок заполняется перед перемещением в `docs/ai/archive/`.

- [ ] Ready for archiving (criteria MVP done)
- Archive status (финальная строка-маркер):
  - `Готово / в архиве: YYYY-MM-DD — docs/ai/archive/FEATURE_<FEATURE_NAME>_CHECKLIST.md`

Пока файл не архивирован, эта “финальная строка-маркер” оставляется пустой/неактивной (или с пометкой “не в архиве”).

## Рекомендуемые пункты Implementation checklist (копируй в фичу)

Ниже — блоки, которые стоит добавлять в `Implementation checklist` конкретной фичи (адаптируй названия файлов под `<feature>`).

### Архитектура и стили (обязательно, если есть `*-form` или `ViewEncapsulation.None`)

- [ ] Форма: на `<form>` (или корневом обёртывающем элементе) задан **уникальный** класс вида `.<feature>-form` (например `.client-form`, `.org-form`).
- [ ] В `*-form.component.scss` при `ViewEncapsulation.None` **все** селекторы вложены под этот корень (`.<feature>-form .field`, не «голый» `.field`). См. `docs/ai/ARCHITECTURE.md` → «Стили форм в features».
- [ ] Нет конфликтов глобального CSS с другими фичами: в браузере открыты **минимум две** страницы (например существующая `/clients` и новая фича) — визуал и цвета не «перетёрты».
- [ ] Новый примитив в **`@ui-kit/*`** или новый **общий** CRUD-класс из `catalog-crud-patterns.scss` отражён в **UI-каталоге** (шапка → **UI**); см. **`docs/ai/UI_CATALOG_AND_DESIGN_SYSTEM.md`**.
- [ ] Соблюдены guardrails: нет `feature → feature`, нет deep-import, UI только `@ui-kit/*` + токены.

### Данные / БЛ (пример)

- [ ] CRUD / фасад в `*.service.ts`, реактивность через `signal` (или согласованный паттерн проекта).
- [ ] Persistence (если MVP): ключ `localStorage` уникален для фичи.

### Данные + HTTP (если фича как `organizations` / `clients`)

Используй **`docs/ai/FEATURE_WITH_API_PATTERN.md`** как чек-лист реализации. В `Implementation checklist` фичи минимум:

- [ ] Модель в **`libs/domain`**, контракт в **`FRONTEND_CONTRACT.md`** (backlog предварительно в **`API_FUTURE_CHECKLIST.md`**).
- [ ] **`core/api/<entity>-api.service.ts`**, экспорт из `core/api/index.ts`, без URL в feature-сервисе кроме как через этот API-сервис.
- [ ] Feature `*Service`: ветка local (`localStorage` + `effect`) vs remote (`getAll`, оптимистичный CRUD, **`listLoading` / `listLoadError` / `mutationError`**, `persistRemoteCache`, fallback при сбое `getAll`).
- [ ] Страница: **`protected readonly`** inject сервиса, баннеры загрузки/ошибок как в **`organizations-page`** / **`clients-page`**, стили `.state-banner` / `.list-state`.
- [ ] **`*.service.spec.ts`**: мок API, сценарии remote success и **getAll fail → кэш**.

### UI / UX (пример)

- [ ] Список, поиск, пагинация, модалки по паттерну существующих фич при необходимости; длинные формы в модалке — см. `ARCHITECTURE.md` → «Модальные окна и формы с многими разделами» (эталон: `organizations`).
- [ ] Если форма "тяжелая" (3+ логических блока), реализация **как в organizations**: вкладки, фиксированная высота/ширина модалки, скролл только внутри панели вкладки, действия снизу.
- [ ] Для сценария `родитель` + quick-add (`+`) проверено каскадное закрытие: сохранение во вложенной форме закрывает только верхнюю модалку.
- [ ] Во вложенных quick-add формах submit изолирован (`(submit)` + `preventDefault/stopPropagation`), родительская форма не получает submit-событие.
- [ ] Кнопка `Сохранить` в родительской форме disabled при невалидных обязательных полях (нет состояния «кнопка активна, но ничего не происходит»).

### Tests & QA

- [ ] `npm run ui:lint`, `nx build web --configuration development`, при наличии тестов — `nx test web`.
- [ ] Ручная проверка сценариев MVP в браузере.
