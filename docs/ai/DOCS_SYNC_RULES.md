# Сторож: синхронизация документации с кодом

## Eve-Arch в коде

При любом изменении, где фигурирует маркер **`Eve-arch: <ARTICLE>`** или семейства **`Eve-BL` / `Eve-UX` / `Eve-PERF` / `Eve-SEC` / `Eve-BE`** (или добавляется новый артикул):

1. Обнови строку в **`EVE_ARCH_INDEX.md`**, в нужном файле из **[`EVE_INDEXES.md`](./EVE_INDEXES.md)** (`EVE_BL_INDEX`, `EVE_UX_INDEX`, …) или в **[`EVE_BACKEND_INDEX.md`](./EVE_BACKEND_INDEX.md)** / `EVE_BE_*.md` для бэка.
2. Обнови **целевой раздел** по **Docs-link** — обычно **`ARCHITECTURE.md#…`**, либо чек-лист фичи / `docs/business/*` для BL.
3. Пример: **`FIXED-LAYOUT-001`** → [`ARCHITECTURE.md`](./ARCHITECTURE.md)`#fixed-layout`; **`UX-ACCESS-003`** → [`EVE_UX_INDEX.md`](./EVE_UX_INDEX.md) + [`ARCHITECTURE.md#a11y-ui-kit`](./ARCHITECTURE.md#a11y-ui-kit).

Если правка **не задаёт повторяемый паттерн**, в коде оставляй **`Eve-arch: 000`** и не раздуй индекс; при необходимости одной строкой отрази нюанс в **`DIAGNOSTIC_CHECKLIST.md`** или чек-листе фичи.

## Eve-BL (бизнес-логика)

При правках **CRUD, статусов, отправки, скидок, цепочек сущностей** — обязательный порядок: **`grep Eve-BL`** → переиспользование или новый артикул → **`EVE_BL_INDEX.md`** → маркер **`Eve-BL:`** в коде. Подробно: **[`EVE_BL_INDEX.md`](./EVE_BL_INDEX.md)** («Правило»).

## Глобальные формулировки («по всему сайту»)

Сначала поиск по репозиторию: **`grep -E "Eve-arch:|Eve-BL:|Eve-UX:|Eve-PERF:|Eve-SEC:|Eve-BE:"`** (или по отдельности), затем согласованно обнови код, индекс и раздел в **`ARCHITECTURE.md`** / чек-листах — см. **`EVE_ARCH_INDEX.md`**, **`EVE_INDEXES.md`**, при бэке — **`EVE_BACKEND_INDEX.md`**.

**Perf / Sec:** артикулы **`PERF-*`** — в [`EVE_PERF_INDEX.md`](./EVE_PERF_INDEX.md); **`SEC-*`** (фронт) — в [`EVE_SEC_INDEX.md`](./EVE_SEC_INDEX.md) (не путать с бэкендом **`Eve-BE:`** / `EVE_BE_SEC.md`).

## Универсальный DoD (для любого нового внедрения)

Причина типовых сбоев (включая мой недочёт): делают только "видимую часть" задачи и пропускают один из слоёв (код, UI-реестр, индексы, валидация).  
Ниже обязательный минимальный DoD — **для любой новой функции/паттерна/темы/экрана**.

Задача считается завершённой только если выполнены **все** пункты:

1. **Код-источник:** изменён реальный источник правды (компонент/сервис/tokens/route), а не только демо/витрина.
2. **Точки подключения:** обновлены места использования (роуты, header actions, DI/providers, feature wiring).
3. **UI/контракт:** если есть пользовательская поверхность (таблица, форма, тема, API-контракт) — обновлён соответствующий реестр/контракт.
4. **Документация по индексу:** обновлены `EVE_*_INDEX.md` + целевой раздел `ARCHITECTURE.md`/чек-листа.
5. **Проверка:** минимум один валидирующий прогон (`nx test ...`, `nx build ...` или явный smoke-сценарий).
6. **Отчёт:** в финале перечислены затронутые файлы и явно написано, что именно внедрено end-to-end.

Если хотя бы один пункт не выполнен, статус задачи — **не "готово"**, а **"частично внедрено"**.

## Темы UI (обязательная полнота внедрения)

При добавлении новой темы (`Лев/Пантера/Орёл/...`) запрещено закрывать задачу, пока не синхронизированы **все** уровни:

1. `design-tokens` (`tokens.scss`) — полный набор переменных темы;
2. header-переключатель в `app.ts` — опция в `<select>` + применение класса на `documentElement`;
3. `localStorage`-восстановление нового значения;
4. таблица цветов `/ui` — заполненная колонка новой темы без пустых строк;
5. `EVE_ARCH_INDEX.md` (`THEME-012`) и раздел темы в `ARCHITECTURE.md`.

## Протокол конфликтов и риска коллапса (обязательно спрашивать)

Если при внедрении есть риск поломки текущего поведения или спорная трактовка требований, исполнитель обязан **остановиться и задать вопрос** до внесения спорных изменений.

Обязательные триггеры для вопроса:

1. Затрагиваются глобальные контейнеры/layout (`app.ts`, shell, header, tokens) и нет явного решения по адаптиву.
2. Изменение может сломать обратную совместимость (API, локальное хранилище, контракты данных).
3. Есть несколько равноправных вариантов реализации и нет приоритета от владельца.
4. Обнаружены неожиданные расхождения между кодом и документацией.

Формат вопроса перед продолжением:
- что конфликтует;
- какие 2-3 безопасных варианта;
- какой вариант предлагается как дефолт и почему.

### Обязательное уведомление о конфликте "доки vs практика"

Если в ходе задачи обнаружено, что:
- текущие правила в docs требуют один подход;
- а стабильная реализация в коде требует другой (временный или fallback),

исполнитель обязан в том же цикле задачи:
1. Отдельно уведомить владельца (краткий блок "Конфликт выявлен").
2. Явно отметить масштаб: локально/глобально, какие экраны затронуты.
3. Записать компромисс в docs (как временное правило) и условия возврата к целевому варианту.
4. В финальном отчёте дать статус: "конфликт закрыт временным решением" или "конфликт снят, возвращён целевой паттерн".

## Если "в коде есть, в UI нет"

Обязательный шаг №1 перед любой глубокой диагностикой:

1. **Один раз перезапустить локальный dev-сервер** (`nx serve ...`) и сделать hard refresh в браузере (`Ctrl+F5`).
2. Только если после этого проблема воспроизводится — идти в анализ кода/роутов/кэша.

Это защищает от ложных "регрессий" из-за старого бандла, зависшего watch-процесса или кэша страницы.

### Диагностика 9: пустое тело dialog при открытии формы

Симптом:
- модалка открывается (overlay/panel видны), но `body` визуально пустой;
- в логах есть `init` формы, но поля/кнопки не видны;
- API/сеть без ошибок (`200`), а контент не появляется.

Проверка:
1. Убедиться, что после перезапуска сервера и `Ctrl+F5` симптом воспроизводится.
2. Быстро проверить, что проблема не в данных (форма инициализируется) и не в API (нет `4xx/5xx`).

Целевой метод (правильный вариант):
1. В первую очередь восстанавливать единый переиспользуемый `app-dialog` (`mode="content"`/`mode="confirm"`) и slot-проекцию `slot="body"`.
2. После фикса обязательно проверить 3+ циклов `open/close` и сценарий `save/cancel`.

Рабочий fallback-метод (только как аварийный):
1. Для проблемного экрана заменить content-диалог на явную разметку:
   - backdrop: `.modal-backdrop`
   - panel: `.modal-panel` (`--md/--lg/--xl` по задаче)
   - форма как прямой дочерний узел panel (без slot-проекции)
2. Проверить smoke-сценарий: открыть/закрыть модалку 3+ раза, заполнить поля, `save/cancel`.
3. Зафиксировать изменение в отчёте как "fallback для устойчивого рендера модалки".

Примечание:
- fallback допустим как временная стабилизация, но правильный финал — возврат к единому `app-dialog`;
- если позже причина в `app-dialog`/проекции будет локализована и безопасно исправлена, можно вернуть единый диалоговый компонент.

Статус на 2026-03-21:
- корневая причина локализована (дублированная проекция `ng-content` по веткам `confirm/content` в `dialog.component.html`);
- применён фикс с единственной точкой проекции тела диалога;
- рабочие экраны возвращены на целевой вариант `app-dialog` для content-модалок.

## Шаблон финального чек-листа (в каждый итоговый ответ)

Использовать как обязательный блок в конце задачи:

```md
## DoD-check (6/6)
- [ ] 1) Код-источник обновлён (не только демо/витрина)
- [ ] 2) Точки подключения обновлены (routes/header/providers/wiring)
- [ ] 3) UI/контракт синхронизирован (таблицы, формы, API/док-контракт)
- [ ] 4) Индексы и архитектурные docs обновлены (`EVE_*_INDEX`, `ARCHITECTURE`)
- [ ] 5) Проверка выполнена (`nx test/build` или smoke) и результат зафиксирован
- [ ] 6) Отчёт содержит список изменённых файлов и статус внедрения (полностью/частично)
```

Правило интерпретации:
- `6/6` = "внедрение полное";
- `<6/6` = "частичное внедрение", обязательно указать оставшиеся шаги и риски.

**Merge / архитектурные правки:** не закрывать PR/задачу как «готово» при `<6/6`. Любое изменение устойчивого архитектурного паттерна сопровождается обновлением **`EVE_*_INDEX.md`** и релевантного раздела **`ARCHITECTURE.md`** (или целевого чек-листа фичи).

**CI docs-gate:** для PR в `main` действует автоматическая проверка (`.github/workflows/docs-gate.yml`). Если меняются `apps/**` или `backend/**`, PR обязан содержать изменения в `docs/**` (или в `.cursor/rules/**` для обновления AI-процесса).

**CI docs-links:** для изменений документации действует проверка ссылок (`.github/workflows/docs-links.yml`) по `docs/**/*.md`.

## Временные решения (обязательная маркировка)

Любой код или док, оставленный как временный (fallback, заглушка, обход бага):

1. **Причина** — один абзац, ссылка на issue/задачу при наличии.
2. **Scope** — какие маршруты/файлы затронуты.
3. **Владелец** — кто отвечает за снятие.
4. **Дата удаления** — целевая дата или условие («после фикса X в ui-kit»).

Без этих четырёх пунктов временное решение считается **недопустимым долгом** и должно быть либо доведено до целевого паттерна, либо явно эскалировано.

## Футер в документах

В конце каждого `.md` под `docs/` — блок **«Синхронизация»** со ссылкой на этот файл, на **`EVE_ARCH_INDEX.md`** и при необходимости на **`EVE_INDEXES.md`** (относительные пути от каталога файла).
