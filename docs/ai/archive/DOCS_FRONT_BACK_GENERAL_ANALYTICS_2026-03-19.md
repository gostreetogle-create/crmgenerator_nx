# Генеральная аналитика документации (frontend + backend)

Дата: 2026-03-19  
Область: единая документация проекта, точка входа, операционность для AI и команды.

## 1) Текущее состояние (что уже хорошо)

- Есть сильный архитектурный базис: `ARCHITECTURE`, `AI_PROJECT_PLAYBOOK`, `DOCS_SYNC_RULES`.
- Разделены бизнес, API и инженерные правила (`docs/business`, `docs/api`, `docs/ai`).
- Есть контур “истина vs backlog” по API (`FRONTEND_CONTRACT` и `API_FUTURE_CHECKLIST`).
- Есть чеклисты фич и архив выполненных работ, включая аналитический лог фиксов.
- Есть project rule для Cursor (`.cursor/rules/crmgenerator-ai.mdc`).

Вывод: фундамент зрелый и пригоден для шаблонизированного масштабирования.

## 2) Главные разрывы до “почти авто-режима”

1. Не было явной единой стартовой страницы для всех ролей и типов задач.
2. Не было отдельного глобального AI-подсказчика `alwaysApply: true`.
3. Не было персонального операционного чеклиста “как перенести стандарт в новый проект Cursor”.
4. Маршрут “что читать первым/вторым” был распределен по нескольким файлам.

## 3) Что добавлено этим пакетом

- `docs/START_HERE.md` — единая точка входа для frontend/backend.
- Обновлен `docs/README.md` с явным приоритетом стартового файла.
- `.cursor/rules/crmgenerator-global-prompts.mdc` — глобальный rule (always apply).
- Обновлен `.cursor/rules/crmgenerator-ai.mdc` (единый вход + корректный путь к prompts).
- `docs/tigran/CURSOR_NEW_PROJECT_PROMPT_CHECKLIST.md` — чеклист переноса стандарта в новый проект.

## 4) Оценка зрелости после изменений

| Направление | До | После | Комментарий |
|---|---|---|---|
| Единая точка входа docs | Частично | Да | `docs/START_HERE.md` |
| Онбординг AI через rule | Частично | Да | global + project `.mdc` |
| Повторяемость для нового проекта | Низкая | Высокая | чеклист для Тиграна |
| Связка frontend/backend docs | Средняя | Высокая | единый маршрут чтения |

## 5) Что еще докрутить до “полировки”

- Добавить в CI простую проверку, что `docs/START_HERE.md` содержит рабочие ссылки на ключевые документы.
- Добавить “release-docs gate”: если меняются `apps/*` или `backend/*`, проверять обновление минимум одного релевантного doc-файла.
- Сформировать 2-3 готовых enterprise-шаблона стартовых проектов (small/medium/large) с одинаковой doc-структурой.
- Добавить авто-шаблон генерации `FEATURE_<NAME>_CHECKLIST.md` командой/скриптом.

## 6) Финальный вывод

Документация приведена к модели “одна точка входа + предсказуемая маршрутизация”.  
Текущего состояния достаточно, чтобы команда и AI работали почти в автоматическом режиме по шаблону, без потери контекста между frontend и backend.
