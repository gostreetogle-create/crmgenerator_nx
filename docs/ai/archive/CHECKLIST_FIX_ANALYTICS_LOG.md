# Аналитика исправлений по чеклистам

Цель файла: фиксировать **каждую исправленную проблему** в формате "что сломалось -> почему -> чего не хватало в документации -> что добавить в правила", чтобы снижать повторяемость ошибок.

Период: 2026-03-21 (полировка + фаза каталога)

---

## Формат записи (обязательный)

1. **Короткое логическое название** (2-6 слов)
2. **Симптом**
3. **Что было причиной**
4. **Почему приняли именно такое решение**
5. **Чего не хватало в доках изначально**
6. **Что добавили/обновили в доках**

---

## Реестр проблем и решений

### 1) Конфликт проекции контента модалки
- **Симптом:** content-модалки открывались с пустым body при рабочем overlay.
- **Причина:** дублированная `ng-content` проекция тела в ветках `confirm/content` в `app-dialog`.
- **Почему так исправили:** один общий блок проекции исключает конфликт Angular content projection и снимает расхождение поведения между режимами.
- **Чего не хватало в доках:** явного запрета на дублирование slot/body-проекции в `ui-kit` диалоге.
- **Что добавили:** критерии регресса и верификацию для `app-dialog` в `ARCHITECTURE.md` + диагностический блок в `DOCS_SYNC_RULES.md`.

### 2) SSR-доступ к `document` в focus-trap
- **Симптом:** prerender падал из-за `document is not defined`.
- **Причина:** browser-only API вызывался без защиты окружения.
- **Почему так исправили:** guard `typeof document !== 'undefined'`/browser check безопасен и не меняет runtime UX в браузере.
- **Чего не хватало в доках:** конкретного SSR-safe правила для директив, работающих с DOM.
- **Что добавили:** акцент на SSR-safe проверках в полировочном трекере и `DOCS_SYNC_RULES.md`.

### 3) Шум отладочных логов в production-коде
- **Симптом:** `console.log` в формах/страницах мешал диагностике и засорял вывод.
- **Причина:** отсутствие жёсткого финального этапа "debug cleanup" перед закрытием задачи.
- **Почему так исправили:** удаление логов снижает шум и убирает ложные сигналы при сборке/SSR.
- **Чего не хватало в доках:** чёткой проверки "в фичах нет debug-log" как отдельного gate.
- **Что добавили:** фиксация пункта как обязательного в трекере полировки (`D1.2`) и итоговом отчёте.

### 4) NG8113: неиспользуемые standalone imports
- **Симптом:** предупреждения Angular компилятора (`ButtonComponent/InputComponent/DialogComponent` не используются в шаблоне).
- **Причина:** устаревший список `imports` после эволюции шаблона.
- **Почему так исправили:** удаление неиспользуемых импортов — самый прямой и безопасный способ убрать предупреждение.
- **Чего не хватало в доках:** регулярной проверки соответствия `imports[]` текущему шаблону.
- **Что добавили:** обязательный повторный build после чистки (`D2.1`, `D2.4`) и запись факта в трекере.

### 5) Legacy `.modal-*` после перехода на `app-dialog`
- **Симптом:** в SCSS оставались старые `.modal-backdrop/.modal-panel`, хотя HTML уже на `app-dialog`.
- **Причина:** неполная миграция (разметка обновлена, сопутствующие стили и проверки — нет).
- **Почему так исправили:** удалили legacy-слой и перенесли нужный flex-паттерн в `dialog`/`catalog-form-host`.
- **Чего не хватало в доках:** отдельного шага "post-migration cleanup" для связанных CSS/TS проверок.
- **Что добавили:** уточнение регресс-критерия в `ARCHITECTURE.md` и фиксация post-polish шага в архиве.

### 6) Бэкенд-статус не совпадал с чеклистом (фаза 1)
- **Симптом:** чеклист отмечал backend REST как незавершённый, хотя CRUD уже был реализован.
- **Причина:** рассинхрон между фактическим кодом и статус-документом.
- **Почему так исправили:** верифицировали маршруты/контроллеры и обновили чеклист по факту.
- **Чего не хватало в доках:** обязательного шага "подтверди в коде перед сменой статуса".
- **Что добавили:** уточнение статусов в `FEATURE_CATALOG_PHASE1_CHECKLIST.md` и архивном трекере.

### 7) Отсутствие Excel-import endpoint'ов для справочников
- **Симптом:** в backlog импорт обозначен, но backend endpoint'ов не было.
- **Причина:** фокус был на CRUD, импорт не был доведён до реализации.
- **Почему так исправили:** добавили минимально безопасную реализацию (`multipart file`, parse `.xlsx/.xls`, отчёт `created/failed/errors`).
- **Чего не хватало в доках:** чёткой схемы "минимальный контракт импорта" (формат файла, поле form-data, формат ответа).
- **Что добавили:** обновления в `FEATURE_CATALOG_PHASE1_CHECKLIST.md`, `API_FUTURE_CHECKLIST.md`, `FRONTEND_CONTRACT.md`.

### 8) Клон товара реализован в backend, но не завершён во frontend
- **Симптом:** API `/products/:id/clone` есть, UI-действия не было.
- **Причина:** этапность внедрения (backend-first) без синхронного закрытия UI.
- **Почему так исправили:** добавили кнопку "Копия" + сервисную логику clone в local/remote режимах.
- **Чего не хватало в доках:** отдельной связки "backend done -> ui action done -> checklist done".
- **Что добавили:** закрытие пункта в `FEATURE_CATALOG_PHASE2_CHECKLIST.md` + контракт clone в `FRONTEND_CONTRACT.md`.

### 9) Mount/Functionality оставались строками, не связями
- **Симптом:** товар хранил одиночные строковые поля вместо набора связей.
- **Причина:** MVP-модель не была обновлена после появления `mount-types`/`functionalities` как справочников.
- **Почему так исправили:** перешли на `mountTypeIds[]`/`functionalityIds[]`, синхронизировали фронт, backend mapper и join-таблицы.
- **Чего не хватало в доках:** чёткого решения "single string vs normalized relations" до начала UI.
- **Что добавили:** явные поля в `FRONTEND_CONTRACT.md`, обновлённый статус в `API_FUTURE_CHECKLIST.md`, пункты закрытия в phase 2 checklist.

## Выводы для улучшения архитектурных документов

- Перед задачей на миграцию добавлять mini-DoD: `markup + styles + handlers + docs`.
- Для каждого критичного reusable-компонента (`app-dialog`) вести раздел "типовые регрессы".
- Для чеклистов "фаза N" добавить авто-проверку статуса по фактическим endpoint'ам/роутам перед ручной отметкой.
- Для backlog API фиксировать минимальный контракт даже для "потом": `request`, `response`, `validation`.
