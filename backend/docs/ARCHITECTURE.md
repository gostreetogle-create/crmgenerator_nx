# Архитектура бэкенда (Node/TS, REST)

## Цель
Поддерживать предсказуемый и масштабируемый REST API под ваш фронт (Angular/Nx) так, чтобы:
- контракты с фронтом оставались синхронными и расширяемыми;
- доменная логика не “растворялась” в контроллерах;
- PDF/задачи и файловые операции не блокировали HTTP;
- правила workflow (статусы, запреты редактирования, snapshot) были единообразны.

## Текущий рекомендуемый стек (для старта)
- Язык: TypeScript
- API: Express (REST)
- БД: PostgreSQL
- ORM: Prisma
- Валидация DTO: `zod`
- Генерация PDF: Puppeteer или Playwright
- Background jobs: таблица `pdf_jobs` + воркер (позже можно заменить на BullMQ/Redis, если станет нужно)

> Если вы выберете MongoDB вместо PostgreSQL — архитектура модулей/слоёв останется, меняется только слой репозиториев и схемы.

## Слои и ответственность

### 1) `apps/api` (composition layer)
- Точки входа: `main.ts`, app setup, middleware (CORS, auth placeholder), маршрутизация.
- Никакой доменной бизнес-логики.

### 2) `src/modules/*` (feature modules)
Каждая сущность/ветка API — отдельный модуль:
- `controllers` (HTTP): тонкие обработчики запроса/ответа, валидация входа, маппинг ошибок.
- `services` (use-cases): workflow, разрешения, snapshot, версионирование, бизнес-валидации.
- `repositories` (persistence): доступ к БД (Prisma), только CRUD/запросы.
- `dto` (валидация/контракты): `zod` схемы и типы для входа/выхода.

### 3) `src/domain` (domain layer)
- Чистые типы, доменные сущности/значения и pure-функции.
- Не импортировать Express/Prisma/queue.

### 4) `src/shared` (cross-cutting)
- utils, error helpers, common types, pagination helpers.

### 5) `src/infra` (integrations)
- prisma client
- файловое хранилище (local volume/S3)
- pdf renderer
- очередь/воркер (если используете отдельные процессы)

## Правило потока
`controllers -> services -> repositories -> database`
`services -> domain (если есть pure-логика)`

## Guardrails
- Контроллер не должен напрямую ходить в репозиторий “на каждой строке” — все сценарии делать через `services`.
- `services` не должны импортировать Express типы.
- Весь вход валидируется через `zod` DTO до бизнес-логики.
- Если фронт использует `docs/api/FRONTEND_CONTRACT.md`, то shape ответа и эндпоинты не “придумываются” — сначала перенос в контракт.
- Ошибки всегда возвращать в едином формате (см. ниже).

## Ошибки API (единый формат)
Рекомендуемый ответ:
- HTTP 4xx/5xx
- body: `{ "message": "человекочитаемый текст", "code"?: "string" }`
Фронт парсит `message` и показывает пользователю.

## Workflow, статусы и “не менять прошлое”
Важные доменные правила для КП/документов:
- полный перечень stable `key` статусов и “когда фиксируем/когда создаём версию” — см. `docs/business/WORKFLOW_STATUSES.md`;
- статусы заранее создаются в БД и завязаны на стабильный `key/statusId`; `name` можно редактировать;
- документ после перехода в “фиксирующий” статус (по текущим договорённостям: `proposal_paid`) переводится в режим “immutable” для `edit/delete`;
- snapshot создаётся в момент фиксации и используется для PDF (и для повторного предпросмотра).

Дополнительно:
- если нужны правки после `proposal_paid`, делается **создание новой версии**, а не “откат” уже фиксированной версии назад.

## PDF генерация (не блокировать сайт)
- HTTP endpoint создаёт job и возвращает `jobId` (202).
- Воркер генерирует PDF, сохраняет файл и помечает job “done”/“failed”.
- API отдаёт ссылку/статус по `jobId`.
- Кэширование: при повторной генерации того же snapshot не пересчитывать.

## Как “вести” документацию
- При добавлении нового API-эндпоинта сначала фиксировать в `docs/api/API_FUTURE_CHECKLIST.md`.
- После согласования — переносить в `docs/api/FRONTEND_CONTRACT.md` и отражать в `libs/domain`.
- Для UI-фич — `docs/ai/FEATURE_<NAME>_CHECKLIST.md`.

## Вопросы для следующего шага (ответьте, и я продолжу заполнять документы)
1. ID в БД: рекомендуем использовать `uuid` (удобно для клонов, snapshot'ов и ссылок между сущностями без “зависимости” от последовательности).
2. PDF хранилище: для MVP достаточно локального volume/папки на сервере; позже можно заменить на S3-совместимое хранилище (S3/MinIO) без изменения API (только изменится backend-реализация слоя хранения).
3. Snapshot КП:
   - вариант A (таблицы): отдельные строки `proposal_item_snapshots`/`proposal_item_snapshot_mounts` и т.п. проще валидировать и искать;
   - вариант B (JSON): один JSON-поле snapshot'а быстрее реализовать для MVP;
   - рекомендуемый “компромисс” для MVP: хранить ключевые поля снимка отдельными колонками (для сумм/проверок/поиска) + остальное как JSON.
4. “Откат” из дочернего к родителю:
   - рекомендуем не делать “статусный откат” назад из `proposal_paid` (чтобы не ломать неизменность прошлого);
   - вместо этого дать действие на уровне “версий/клонов”: отменить дочернее КП (или пометить `canceled`) и/или создать новую “заменяющую” версию, а родитель остаётся неизменным.
